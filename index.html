The Typical Rails File Structure

  • app/ - contains your assets, helpers, mailers, controllers, model and views files. This is where you will spend close to 75% of your time coding.

  • bin/ - contains your basic ruby scripts. This includes the commands that we run in the terminal such as bundle, rake and rails.
  • config/ - contains the major configurations needed for the app. This includes the three environments:
    • development.rb
    • production.rb
    • test.rb
    • application.rb
    routes.rb for creating routes and database.yml for database configuration.
  • db/ - contains the seeds.rb file which is use to create the application default data. When you begin to make and add new changes changes to the database a migrate folder is created along with the schema.rb, which shows the current state of the app's database.
  • lib/ - contains reuseable library code that you write. Typically left empty except for the assets and tasks folder.

  • log/ - contains the data that on the app when it is being run in the development, production or test environment.

  • public/ - contains the files that are sent to the user as if the files were in the root directory. Case example would be error messages like 404.html or 500.html. This also contains favicon.ico and robots.txt files.
  • test/ - contains all of the subdirectories with automated tests for each part of your application.

  • tmp/ - contains ephemeral files. This is where you finde cached copies of your app's assets, process ids, user sessions and files representing sockets being used in the app.

  • vendor/ - contains assets that are needed by third-party gems whenever you add gems to your application.

Constructing the Controller

R.E.S.T

Representational State Transfer is a client-server software architecture introduced in 2000 by Dr. Roy Fielding, one of the authors of HHTP specification. The REST architecture deals with representation of resources.

  1. Client initiates a request to the server
  2. Server processes that request
  3. Server returns a response to the Client

All of this is done in the Rails controller

Global State

Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the page background.

State Events

Additionally custom events can be triggered on a per slide basis by binding to the data-state name.
